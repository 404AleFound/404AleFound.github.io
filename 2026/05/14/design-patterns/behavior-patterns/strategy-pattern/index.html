Strategy Pattern

2026-05-14 · updated 2026-05-19 · design-patterns , behavior-patterns

Note

Strategy pattern will define a family of algorithms, encapsulate each into its own class, and make them interchangeable — so the caller never needs to know the details.

1 Background

Imagine you’re building a navigation app. It starts simple — just driving routes. Then you add walking, transit, cycling… Each new algorithm bloats the main class a little more. Fix one bug, the whole class shakes. Add a feature, merge conflicts everywhere.

That’s exactly the problem Strategy Pattern was designed to solve.

2 Solution: Strategy Pattern

Strategy pattern will define a family of algorithms, encapsulate each into its own class, and make them interchangeable — so the caller never needs to know the details.

There are three layers in the Strategy pattern:

  • the Context, which holds a reference to a strategy and delegates work to it without knowing the details;
  • the Strategy Interface, a single shared contract that all algorithms implement;
  • the Concrete Strategies, each encapsulating one algorithm independently — swappable at any time by the client.

3 Implement

  • STEP 1: Find the algorithm in the context class that changes frequently — often a big if/else or switch.
  • STEP 2: Declare a common strategy interface shared by all algorithm variants.
  • STEP 3: Extract each algorithm into its own concrete class, all implementing that interface.
  • STEP 4: In the context class, add a field to hold a strategy reference and a setter to swap it out.
  • STEP 5: The client picks the right strategy and injects it into the context.

5 Analysis

Use Strategy when:
  1. You need to switch algorithms at runtime.
  2. You have many similar classes differing only in behavior.
  3. You want to isolate business logic from algorithm details.
  4. A massive if/else is switching between algorithm variants.

Advantages:
  1. Swap algorithms at runtime freely
  2. Decouples algorithm implementation from the caller
  3. Favors composition over inheritance
  4. Open/Closed Principle — add new strategies without touching Context

Disadvantages:
  1. Overkill if you only have a few stable algorithms
  2. Clients need to know the differences between strategies to pick the right one
  3. Modern languages can use lambdas instead — no extra classes or interfaces needed
