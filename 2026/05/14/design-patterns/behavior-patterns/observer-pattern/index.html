Observer Pattern

2026-05-14 · updated 2026-05-19 · design-patterns , behavior-patterns

Note

Observer is a behavior pattern design pattern that lets you define a subscription mechanism to notify multiple objects about any events that happen to the object they’re observing.

1 Background

Imagine that you have two types of objects: a Customer and a Store. The customer is very interested in a particular brand of product which should become available in the store very soon.

To solve the problem, here are the two solutions:

  • Sol1: The customer could visit the store every day and check product availability.
  • Sol2: The store could send tons of emails to all customers each time a new product becomes available.

For solution 1, the customer will waste time checking product availability. While for solution 2, the store will waste resources notifying the wrong customers.

Observer pattern can help to solve the conflict.

2 Solution: Observer Pattern

Before, some concepts should be defined. The object that has some interesting state, such as Store, is often called subject or publisher, since it also need to notify other objects. All other objects that want to track changes to the publisher’s state are called subscribers.

The subject or publisher always need a subscription mechanism, shown below:

  • An array field for storing a list of references to subscriber objects.
  • Several public methods which allow adding subscribers to and removing them from that list.

Now, whenever an important event happens to the publisher, it goes over its subscribers and calls the specific notification method on their objects.

However, in real apps development, there may be dozens of different subscriber classes. So it’s crucial that all subscribers implement the same interface and that the publisher communicates with them only via that interface.

To make all publishers follow the same interface, all the concrete subscriber classes should inherit from the same interface.

3 Implement

  • STEP 1: Split business logic into a publisher (core functionality) and subscribers (reactive logic)
  • STEP 2: Declare a subscriber interface with at least one update method.
  • STEP 3: Declare a publisher interface with subscribe / unsubscribe methods — publishers interact with subscribers only through the subscriber interface.
  • STEP 4: Put subscription list and its management into an abstract base publisher class.
  • STEP 5: Implement concrete publisher classes that notify all subscribers when something significant happens.
  • STEP 6: Implement update in concrete subscriber classes, receiving event context as a parameter; alternatively, pass the publisher itself so subscribers can pull data directly.
  • STEP 7: The client creates all subscribers and registers them with the appropriate publishers

4 Analysis

Advantages:

  • Open/Closed Principle. You can introduce new subscriber classes without having to change the publisher’s code (and vice versa if there’s a publisher interface).
  • You can establish relations between objects at runtime.
    Disadvantages:
  • Subscribers are notified in random order.
