Decoretor Pattern

2026-05-14 · updated 2026-05-19 · design-patterns , structural-patterns

Note

Decorator is a structural design pattern that lets you attach new behaviors to objects by placing these objects inside special wrapper objects that contain the behavior without modifying it.

1 Background

Imagine you’re building a notification library. It starts with a single Notifier class that sends emails. Simple enough.

Then users want SMS. Then Facebook. Then Slack. You reach for inheritance:

Still manageable. But then someone asks: “What if I want email and SMS and Slack at the same time?”

You start creating combination subclasses:

The amount of subclasses increase from 3 to 15. Add one more channel and it doubles. This is the combinatorial explosion problem.

This is because inheritance is static. Which channels to use is a runtime decision — inheritance can’t handle dynamic combinations.

You have to find some other way to structure notifications classes so that their number won’t accidentally break some Guinness record.

2 Solution: Decoretor Pattern

Extending a class is the first thing that comes to mind when you need to alter an object’s behavior. However, inheritance has several serious caveats that you need to be aware of:

  • Inheritance is static. You can’t alter the behavior of an existing object at runtime. You can only replace the whole object with another one that’s created from a different subclass.
  • Subclasses can have just one parent class. In most languages, inheritance doesn’t let a class inherit behaviors of multiple classes at the same time.

3 Implement

  1. Confirm your domain fits the pattern — a core component with optional add-on layers.
  2. Define a shared interface covering methods common to both the component and decorators.
  3. Implement the concrete component with the base behavior.
  4. Create a base decorator that holds a reference to the wrapped object (typed as the interface) and delegates all calls to it.
  5. Extend the base decorator into concrete decorators, each adding behavior before or after delegating to the wrapped object.
  6. Let the client compose decorators freely at runtime.
